# Стилист — план продукта: от идеи до запуска

> AI-помощник, который определяет типаж человека (система Кибби + цветотип по фото), подбирает аксессуары и даёт советы по цветовой палитре. Документ описывает стратегию, архитектуру, команду AI-агентов и пошаговый план реализации.

---

## 1. Суть продукта и для кого он

Пользователь приходит с запросом «помоги понять, что мне идёт». Продукт ведёт его через короткий диалог: задаёт вопросы о фигуре и предпочтениях, просит загрузить несколько фото, анализирует их и выдаёт три вещи — **типаж по Кибби**, **цветотип с палитрой** и **персональные рекомендации** (силуэты, ткани, аксессуары, что покупать и чего избегать). Дальше пользователь может задавать уточняющие вопросы в чате, как живому стилисту.

Ядро ценности — не «ещё один тест из интернета», а связка «анализ фото + экспертная база знаний + диалог». Основная аудитория: женщины 20–45, которые интересуются стилем, но не готовы платить 5–15 тыс. ₽ за очную консультацию стилиста. Вторичная — мужская аудитория и профессиональные стилисты (как рабочий инструмент).

## 2. Рекомендация по платформе

Ты пока не выбрал платформу — рекомендую **начать с веб-приложения, упакованного как Telegram Mini App (плюс обычный PWA-доступ через браузер)**.

Почему именно так на старте:

- **Дёшево и быстро.** Один кодовый веб-фронтенд работает и внутри Telegram, и в браузере. Не нужно проходить ревью App Store / Google Play на каждой итерации.
- **Загрузка фото и чат — родные сценарии Telegram.** Пользователь уже залогинен, не нужно строить регистрацию и присылать SMS.
- **Готовая дистрибуция и оплата.** В Telegram есть встроенные платежи и простой механизм шеринга — важно для роста на стадии «только идея».
- **Переиспользование.** Когда продукт подтвердит спрос, тот же код легко завернуть в нативное приложение (через Capacitor/React Native WebView) — без переписывания с нуля.

Нативное iOS/Android-приложение имеет смысл во **второй фазе**, когда появятся push-уведомления, оффлайн-гардероб и подписка через сторы. Делать его сразу — это лишние месяцы и деньги до проверки гипотезы.

## 3. Главный технический вопрос: можно ли определить Кибби по фото

Здесь нужна честность, потому что от этого зависит весь продукт.

**Кибби — субъективная система.** Даже сертифицированные стилисты-эксперты нередко расходятся в типировании одного и того же человека. «Чистый» компьютерный алгоритм (просто CV-модель → один из 13 типов) надёжно работать не будет: типаж зависит от совокупности признаков (костяк, вертикаль, «плоть» тела, черты лица), которые трудно измерить по одному снимку.

**Рабочий подход — гибрид, а не магия одной кнопки:**

1. **Vision-модель (мультимодальный LLM с поддержкой изображений)** анализирует 2–4 фото (в полный рост спереди/сбоку + лицо) и оценивает признаки по шкале «инь ↔ ян»: ширина и тип костяка, длина вертикальной линии, мягкость/острота черт лица, характер «плоти» тела.
2. **Структурированный опросник** добирает то, что фото не покажет: рост (ключевой параметр Кибби!), ощущение пропорций, как сидит одежда, прошлый опыт.
3. **Скоринговая модель** сводит признаки в баллы по параметрам Кибби и предлагает **тип + уровень уверенности** («скорее всего Soft Classic, 72%; альтернатива — Soft Natural»).
4. **Пользователь подтверждает или корректирует** результат. Это и повышает точность, и снимает с продукта ответственность за «истину в последней инстанции».

Позиционирование внутри продукта: **«умный ассистент-стилист», а не оракул.** Цветотип по фото определяется надёжнее Кибби (анализ подтона кожи, контраста, цвета глаз/волос) — его можно подавать увереннее.

## 4. Архитектура системы

```
┌─────────────────────────────────────────────────────────┐
│  Клиент: Telegram Mini App / PWA (React)                  │
│  онбординг · опросник · загрузка фото · чат · результаты  │
└───────────────────────────┬─────────────────────────────┘
                            │ REST / WebSocket
┌───────────────────────────▼─────────────────────────────┐
│  Backend API (FastAPI / Python)                           │
│  auth · сессии · хранение согласий · оркестрация          │
└───────┬─────────────────┬──────────────────┬────────────┘
        │                 │                  │
┌───────▼──────┐  ┌───────▼───────┐  ┌───────▼──────────┐
│ Vision-слой   │  │ Слой знаний    │  │ Движок            │
│ анализ фото:  │  │ (RAG):         │  │ рекомендаций      │
│ цвет, лицо,   │  │ цвет (12       │  │ наложение 6 линз: │
│ фигура,       │  │ сезонов),      │  │ палитра, силуэты, │
│ пропорции,    │  │ Кибби, фигура, │  │ ткани, вырезы,    │
│ Кибби         │  │ лицо, архетипы │  │ аксессуары, товары│
└──────────────┘  └───────────────┘  └───────┬──────────┘
                                            │
            ┌────────────────────┐  ┌────────▼───────────┐
            │ Фид трендов          │  │ Каталог товаров /   │
            │ Pantone / WGSN-стиль │─►│ партнёрские ссылки  │
            │ (обновляется сезонно)│  │                     │
            └────────────────────┘  └────────────────────┘

Хранилища: Postgres (пользователи, сессии, результаты) ·
S3-совместимое объектное хранилище (фото, с шифрованием и TTL)
```

**Принципиально важно — приватность фото.** Лицо и тело человека — чувствительные биометрические данные. Нужны: явное согласие на обработку, возможность удалить фото в один клик, автоудаление по таймеру (например, через 30 дней), хранение в зашифрованном виде. Это требование и закона (152-ФЗ в РФ, GDPR в ЕС), и доверия пользователей.

**Рекомендуемый стек:** React + Telegram SDK (фронтенд) · FastAPI/Python (бэкенд) · мультимодальный LLM с vision (анализ фото и диалог) · векторная БД (Qdrant/pgvector) для базы знаний по стилю · Postgres · S3/MinIO для фото. Платежи и push — во второй фазе.

## 5. Команда AI-агентов

Здесь два уровня, и оба важны.

### 5.1. Агенты, которые помогают РАЗРАБОТАТЬ продукт

Это «виртуальная команда» под твоим руководством — каждому агенту даётся роль, контекст и зона ответственности:

| Агент | Роль | Что делает |
|---|---|---|
| **Product Lead** | Продакт-менеджер | Держит видение, режет скоуп до MVP, приоритизирует, пишет требования |
| **Style Expert** | Эксперт-стилист | Формализует все методы (цвет, Кибби, фигура, лицо, архетипы, тренды) в базу знаний и логику скоринга — см. `Методология_стилиста.md` |
| **CV/ML Engineer** | Инженер компьютерного зрения | Проектирует пайплайн анализа фото, промпты для vision-модели, скоринг по каждому слою |
| **UI/UX Designer** | Дизайнер интерфейса и опыта | Структура экранов, прохождение пользователя, макеты, визуальный язык, удобство загрузки фото и подачи результата |
| **Frontend Engineer** | Фронтенд-разработчик | Реализует макеты дизайнера в коде Mini App, верстает, подключает к API |
| **Backend Engineer** | Бэкенд | API, оркестрация агентов, хранение, согласия и приватность, фид трендов |
| **UX/Content Writer** | Контент и тон диалога | Тексты бота, формулировки советов, онбординг, деликатные темы (тело, внешность) |
| **QA / Critic** | Тестировщик и «адвокат дьявола» | Проверяет точность типирования, ищет ошибки, тестирует крайние случаи |

**Нужен ли отдельный UI/UX Designer или это входит во frontend?** Это две разные роли, и на проекте с фокусом на «лёгкий, человечный диалог» их лучше разделить. **UI/UX Designer решает, ЧТО и в каком порядке видит пользователь** (логика экранов, путь от первого вопроса до результата, как мягко попросить фото, как подать чувствительный результат о фигуре) — это проектирование опыта. **Frontend Engineer решает, КАК это собрать в работающем коде** (вёрстка, анимации, подключение к бэкенду). Совмещение в одном человеке/агенте обычно приводит к тому, что страдает либо продуманность пути, либо качество реализации. Для нашего продукта, где UX деликатность критична, я добавил дизайнера отдельной ролью.

На практике в этой среде ролями управляешь ты, а я выступаю исполнителем каждой из них по очереди (или через под-агентов под конкретную задачу). Начинать «нанимать» отдельных под-агентов стоит, когда задача станет действительно параллельной — например, дизайнер рисует макеты, пока Style Expert наполняет базу знаний.

### 5.1.1. Про подключение Claude / vision-API к онлайн-ресурсу

Ты предложил подключить API к онлайн-ресурсу для анализа — это правильное направление, и финально способ выберет команда (роли CV/ML + Backend). Чтобы решение было осознанным, вот рамка выбора, которую они будут применять:

- **Анализ фото** выполняет мультимодальная модель с поддержкой изображений (vision-LLM) через API. Это и есть «онлайн-ресурс для анализа» — мы не обучаем свою CV-модель с нуля на старте, а вызываем готовую через API и направляем её точными промптами под каждый слой (цвет, лицо, пропорции).
- **Диалог и рекомендации** — текстовая LLM через API + база знаний (RAG) по методологии.
- **Решения, которые принимает команда:** какой именно провайдер модели; где проходит граница «модель vs детерминированный скоринг» (часть выводов, например замер пропорций, дешевле и стабильнее считать алгоритмически, а не моделью); как кэшировать результаты ради стоимости; где хранить фото с учётом приватности при отправке во внешний API.
- **Принцип:** внешний API — это «глаза и язык» продукта, а наша ценность — в **методологии и оркестрации поверх него** (как именно мы складываем 6 линз в один профиль). Это то, что нельзя скопировать одним API-вызовом.

### 5.2. Агенты ВНУТРИ продукта (рантайм)

Это то, что реально работает «под капотом» бота, когда им пользуется человек:

1. **Агент-интервьюер** — ведёт диалог, собирает опросник (рост, лайфстайл, цели, архетип через выбор образов), просит нужные фото в нужных ракурсах.
2. **Агент-аналитик фото** — извлекает из снимков параметры всех «фото-слоёв»: цветотип (подтон, светлота, насыщенность, контраст), форму лица, тип фигуры и пропорции, признаки Кибби.
3. **Агент-типолог** — сводит фото-признаки + ответы опросника в единый профиль: цветосезон (из 12), тип фигуры, форма лица, тип Кибби и архетип — каждый с уровнем уверенности.
4. **Агент-трендвотчер** — накладывает на постоянный профиль актуальные цвета и силуэты сезона из обновляемого фида (Pantone / прогноз индустрии), адаптируя тренд под палитру пользователя.
5. **Агент-стилист** — выдаёт рекомендации на пересечении слоёв: палитра, силуэты, ткани, вырезы, аксессуары, капсула, чего избегать — с объяснением «почему».
6. **Агент-консультант** — отвечает на свободные вопросы в чате, опираясь на полный профиль пользователя и базу знаний.

Оркестратор на бэкенде вызывает их по очереди и хранит общий профиль пользователя как единый контекст. Методология, на которой работают агенты 2–5, подробно описана в `Методология_стилиста.md`.

## 6. Дорожная карта

**Фаза 0 — Прототип и проверка концепции (1–2 недели).**
Интерактивный прототип UX (есть в этом проекте) → показать 10–15 потенциальным пользователям, собрать реакцию. Параллельно — собрать ядро базы знаний по Кибби и цветотипам.

**Фаза 1 — MVP (4–8 недель).**
Telegram Mini App с реальной логикой: опросник + загрузка фото → vision-анализ → результат (Кибби + цветотип + базовые советы) → чат с консультантом. Согласие на обработку фото и автоудаление. Цель — первые живые пользователи и замер точности.

**Фаза 2 — Рекомендации и удержание (4–6 недель).**
Подбор конкретных товаров со ссылками, сохранение результата, «гардероб», push/напоминания, шеринг результата. Здесь же — первая монетизация.

**Фаза 3 — Масштаб (по результатам).**
Нативное приложение, мужская линейка, инструмент для профи-стилистов, расширение каталога и партнёрств.

## 7. Монетизация (на будущее — пока только намечаем)

Ты отметил, что это «пока идея», поэтому фиксирую варианты без выбора:

- **Freemium-подписка** — базовый разбор бесплатно, расширенный (полный гардероб, неограниченный чат, сезонные капсулы) по подписке. Классика для таких приложений.
- **Разовые покупки** — платный глубокий разбор образа или подбор капсульного гардероба под заказ.
- **Партнёрки с магазинами** — подбор товаров со ссылками на маркетплейсы, комиссия с продаж. Хорошо сочетается с подпиской.

Оптимально позже комбинировать freemium + партнёрку: бесплатный вход для роста аудитории, деньги — с подписки и комиссий.

## 8. Главные риски и как их снять

- **Субъективность Кибби → недовольство точностью.** Снимается позиционированием «ассистент, а не оракул», уровнями уверенности и возможностью скорректировать тип.
- **Приватность фото.** Снимается явным согласием, шифрованием, автоудалением, прозрачной политикой.
- **«Холодный» каталог товаров.** На старте — общие рекомендации по типам вещей; конкретные товары и партнёрки подключать в Фазе 2.
- **Стоимость vision-запросов.** Кэшировать анализ, ограничивать число фото, для бесплатного тарифа — лимиты.
- **Деликатность темы тела/внешности.** Тон диалога продумывается отдельно (роль UX/Content Writer): только поддерживающие, не оценочные формулировки.

## 9. Ближайший следующий шаг

Прототип UX готов (см. файл `prototype` в этом проекте) — он показывает полный путь пользователя: онбординг → опросник → загрузка фото → результат Кибби и цветотип → советы и аксессуары → чат. На нём удобно собрать первую обратную связь и уточнить, что докручивать перед началом разработки MVP.
