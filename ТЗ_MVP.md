# Техническое задание: MVP «Стилист»

Версия 1.0 · документ для команды разработки (AI-агентов и людей)

---

## 1. Цель и границы MVP

**Цель.** Дать пользователю через чат-бота персональный стилевой профиль (цветосезон, типаж Кибби, архетип) и конкретные рекомендации (палитра, силуэты, ткани, вырезы, аксессуары, капсула), опираясь на ответы опросника и — опционально — на анализ фото.

**В рамках MVP:**
- Диалог-опросник, ведущий пользователя к профилю.
- Движок анализа, собирающий профиль из 3 слоёв: **цветосезон (12), Кибби (13 типов), архетип (7)**.
- Развёрнутая **база знаний** по всем типам с рекомендациями.
- HTTP API, к которому подключается любой чат-бот.
- Готовый коннектор к **Telegram**.
- Опциональный приём параметров анализа фото от внешней vision-модели.

**Вне рамок MVP** (фаза 2+): собственная CV-модель, платежи и подписки, подбор конкретных товаров со ссылками, сохранение гардероба, push, нативные приложения, фид трендов реального времени.

---

## 2. Архитектура MVP

```
Пользователь ──► Чат-бот (Telegram) ──► HTTP API ──► Движок анализа ──► База знаний (JSON)
                      │                                   │
                      └── ведёт опросник                  └── скоринг: Кибби, сезон, архетип
                                                              + сборка рекомендаций
        (опционально) Vision-модель ──► параметры фото ──► в движок
```

Принцип: **движок не зависит от канала**. Telegram — лишь один из коннекторов; тот же API можно подключить к веб-виджету, WhatsApp, Mini App. Анализ фото — внешний поставщик параметров, движок принимает уже извлечённые признаки.

---

## 3. Развёртывание базы знаний (ключевое требование)

База знаний — отдельный слой данных в формате JSON, который движок загружает при старте. Должна быть развёрнута **полностью**, по всем типам:

### 3.1. Кибби — все 13 типов
Файл `kb/kibbe.json`. Каждый тип содержит: `id`, название (рус.), семейство (Dramatic / Natural / Classic / Gamine / Romantic), описание линий (баланс инь-ян), рекомендации (`silhouettes`, `fabrics`, `details`, `accessories`, `necklines`), список `avoid`.

Полный перечень: Dramatic, Soft Dramatic, Flamboyant Natural, Natural, Soft Natural, Dramatic Classic, Classic, Soft Classic, Flamboyant Gamine, Gamine, Soft Gamine, Theatrical Romantic, Romantic.

### 3.2. Цветосезоны — все 12
Файл `kb/seasons.json`. Каждый сезон содержит: `id`, название, `undertone` (warm/cool/neutral), `value` (light/medium/deep), `chroma` (bright/medium/soft), `palette` (массив HEX), `neutrals` (HEX), характеристики внешности (`hair`, `skin`, `eyes`, `contrast`), `metals` (золото/серебро), `avoid`.

Полный перечень: Bright Spring, True/Warm Spring, Light Spring, Light Summer, True/Cool Summer, Soft Summer, Soft Autumn, True/Warm Autumn, Deep/Dark Autumn, Deep/Dark Winter, True/Cool Winter, Bright Winter.

### 3.3. Архетипы стиля — 7
Файл `kb/archetypes.json`. Каждый архетип: `id`, название, описание, ключевые слова, сигнатурные вещи, как смешивается с другими.

Полный перечень: Классический, Драматический, Романтический, Натуральный, Элегантный шик, Творческий, Бунтарский.

### 3.4. Требования к базе знаний
- Данные **отделены от кода** — можно править без изменения логики.
- Каждая запись самодостаточна (движок не достраивает текст рекомендаций — берёт из базы).
- Валидируется при загрузке (наличие обязательных полей).
- Расширяема: формат позволяет дописывать поля (например, под тренды) без поломки.

## 4. Движок анализа (`engine.py`)

Чистый Python, без внешних зависимостей. Главная функция:

```
analyze_profile(answers: dict, vision: dict | None = None) -> dict
```

### 4.1. Вход
- `answers` — ответы опросника: рост, костяк/плечи, черты лица, посадка одежды, подтон (вены/реакция на металл), светлота (волосы/глаза/кожа), контраст, предпочтения по образам (для архетипа).
- `vision` (опционально) — параметры от фото-модели: `yin_yang`, `vertical`, `undertone`, `value`, `chroma`, `contrast`. Если переданы — имеют приоритет/усиливают скоринг.

### 4.2. Логика скоринга
- **Кибби.** Суммируются баллы по шкале инь↔ян (костяк, лицо, посадка) и по «вертикали» (рост). Сначала определяется семейство, затем уточняется тип внутри семейства. Возвращается тип + `confidence` (зависит от выраженности баланса).
- **Цветосезон.** Три независимые оси: `undertone` (тёплый/холодный/нейтральный), `value` (светлый/средний/глубокий), `chroma` (яркий/средний/мягкий). Комбинация осей → один из 12 сезонов по таблице соответствий. `confidence` — по согласованности ответов.
- **Архетип.** По ответам о предпочитаемых образах накапливаются веса архетипов; берётся `primary` + `secondary`.

### 4.3. Выход
Единый объект профиля:
```json
{
  "kibbe":   {"id","name","family","confidence","line","recommendations","avoid"},
  "season":  {"id","name","undertone","value","chroma","confidence","palette","neutrals","metals","avoid"},
  "archetype":{"primary","secondary","names","description"},
  "summary": "связный текст профиля",
  "recommendations": {"silhouettes","fabrics","necklines","accessories","capsule","avoid"},
  "disclaimer": "Кибби и архетип — гипотеза, можно скорректировать"
}
```

### 4.4. Требования
- Детерминированность: одинаковый вход → одинаковый выход.
- Граничные случаи: неполные ответы не должны ронять движок (разумные дефолты + снижение confidence).
- Встроенный self-test (`engine.py --selftest`) на нескольких эталонных профилях.

## 5. HTTP API (`api.py`)

Стандартная библиотека Python (`http.server`), без фреймворков — чтобы MVP запускался без установки зависимостей.

| Метод | Путь | Назначение |
|---|---|---|
| `GET` | `/health` | проверка живости |
| `POST` | `/analyze` | принять `answers` (+`vision`), вернуть профиль |
| `GET` | `/questions` | вернуть структуру опросника (для любого клиента) |

- Формат — JSON. CORS включён (для будущего веб-клиента).
- Ошибки — корректный код + JSON с описанием.

## 6. Интеграция с чат-ботом (`telegram_bot.py`)

Готовый коннектор к Telegram на стандартной библиотеке (long polling через `urllib`), чтобы не тянуть зависимости.

- Ведёт пользователя по опроснику кнопками (inline keyboard).
- По завершении вызывает движок (напрямую или через API) и присылает профиль: тип Кибби, сезон с палитрой, архетип, рекомендации.
- Поддерживает свободный вопрос в конце (передаётся консультанту).
- Точка приёма фото предусмотрена как заглушка (`TODO: vision`), готовая к подключению внешней модели.
- Запуск: переменная окружения `TELEGRAM_TOKEN`.

Архитектурно бот **тонкий** — вся логика в движке; заменить Telegram на другой канал = написать новый коннектор тех же 100 строк.

---

## 7. Нефункциональные требования

- **Зависимости:** ноль внешних пакетов для запуска ядра (только стандартная библиотека Python 3.9+). Это сознательное решение MVP ради лёгкого старта; в фазе 2 допускается переход на FastAPI и т.д.
- **Приватность фото:** в MVP фото не хранится сервером; vision-параметры приходят уже извлечёнными. При подключении реального фото-анализа — согласие, шифрование, автоудаление (см. план продукта).
- **Тон ответов:** только поддерживающие, не оценочные формулировки о теле и внешности.
- **Честность результата:** Кибби и архетип всегда сопровождаются дисклеймером и возможностью скорректировать.

## 9. Критерии готовности MVP (Definition of Done)

1. База знаний развёрнута полностью: 13 Кибби + 12 сезонов + 7 архетипов, валидируется при загрузке.
2. `engine.py --selftest` проходит на эталонных профилях, выдаёт связный осмысленный результат.
3. API поднимается командой `python api.py` и отвечает на `/analyze`.
4. `telegram_bot.py` запускается с токеном и проводит полный диалог до выдачи профиля.
5. README объясняет запуск и подключение к чат-боту за 5 минут.
